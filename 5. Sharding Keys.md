# Sharding Keys

A sharding key is a column or expression used to determine which shard (server) a row belongs to in a distributed table.
- It ensures data is distributed across multiple nodes.
- Helps parallelize queries and avoid hotspot nodes.
- Same value always maps to the same shard

## Simple Column Sharding

Use a single column as the sharding key. Each value of the column is hashed, then modulo by shard count.

Example:

```
CREATE TABLE distributed_events
(
    user_id UInt64,
    event_type String,
    event_time DateTime
)
ENGINE = Distributed(
    cluster_name,
    default_db,
    events_local,
    user_id  -- sharding key
);
```

Use case:
- High-cardinality columns, e.g., user_id, session_id.
- Ensures even data distribution when values are fairly uniform.

## Expression-Based Sharding

Use a function or expression of one or more columns. Expression is computed, then hashed to determine shard.

Example:

```
CREATE TABLE distributed_events
(
    user_id UInt64,
    event_type String,
    event_time DateTime
)
ENGINE = Distributed(
    cluster_name,
    default_db,
    events_local,
    cityHash64(user_id, toYYYYMM(event_time))
);
```

Use case:
- Compound keys help distribute data based on multiple columns.
- Example: splitting user_id but also considering event_time to prevent time-based hotspots.

## Random Sharding

Assign rows to shards randomly. Use rand() or constant modulo expression as the sharding key.

Example:

```
CREATE TABLE distributed_events
(
    user_id UInt64,
    event_type String
)
ENGINE = Distributed(
    cluster_name,
    default_db,
    events_local,
    rand()  -- random sharding
);
```

Use case:
- No natural high-cardinality column exists.
- Ensures roughly even distribution but hard to target specific shards.

## Modulo-Based Sharding

Takes modulo by some number, typically the number of shards, then maps row to shard.

Example:

```
CREATE TABLE distributed_events
(
    user_id UInt64,
    event_type String
)
ENGINE = Distributed(
    cluster_name,
    default_db,
    events_local,
    user_id % 4  -- if 4 shards
);
```

Use case:
- Useful when column values are already integers.
- Simple, predictable mapping.
- Can lead to imbalance if column values are not uniformly distributed.

### What happens if number of shards changes?

Shard assignment depends on modulo of hash / N. If N (number of shards) changes, most of the existing data will map to different shards.

| Scenario           | Outcome                                                                                                                                                  |
| ------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Adding a new shard | Existing data stays on old shards → new data may go to new shard. Queries may need **resharding** to balance old and new data                            |
| Reducing shards    | Some shards are effectively ignored → data might become inaccessible unless redistributed                                                                |
| Queries            | `Distributed` table will route rows according to the **current shard expression**, so inconsistent shard count can lead to missing or duplicated results |
